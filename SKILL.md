# 多 Agent 互动短剧生视频

用于把用户的题材、爽点、角色设定或故事种子，制作成可选择分支的互动短剧；先让用户选择或确认题材，再生成分支树并用 ChatGPT/OpenAI 图像生成可视化分支图，随后生成角色定妆图并给用户确认角色外观，最后通过可切换视频生成 provider 生成各互动节点的视频片段。

默认形态：中文短剧/短视频、分支剧情、主 agent 统筹、多 agent 并行评审、先角色定妆、后视频生成。

## 前置要求

分支可视化图和角色图阶段必须加载 `imagegen` skill，默认使用内置 `image_gen` 工具生成图片；不要使用 Crate 或其他视频平台的生图模型做分支可视化或角色定妆，除非用户明确要求改用 Crate 生图。

视频阶段默认先选择 provider，而不是硬绑定单一平台。性价比优先时，优先考虑 fal.ai / Replicate 上的 Wan、Hailuo、Kling 等图生视频模型；需要精修时再考虑 Kling 官方、Runway、Luma；用户明确指定 Crate 时再使用 Crate CLI。

生成命令会消耗账号 credits。提交真实生成前，必须先展示生成清单、预计提交节点、provider、模型、预计成本或计费单位，并明确告知即将消耗 credits。

如果使用 Crate CLI，以本机 CLI 帮助和实际服务端返回为准：

```bash
crate --help
crate doctor --json
crate model list --type image-to-video --json
crate generate --help
crate generate submit --help
crate task --help
crate upload file --help
crate task get <taskId> --json
```

题材选择是剧本生成前的硬门槛：未让用户按背景、主题、设定三个维度选择或确认题材前，不要写正式剧本、分支树、角色图 prompt 或视频 prompt。

分支可视化是进入角色定妆前的默认步骤：生成分支树后，必须用生图模型生成一张分支可视化图并展示给用户。用户明确说“跳过可视化”时才可以跳过。

角色图确认是硬门槛：未向用户展示角色图并获得确认前，不要提交视频生成任务。

角色图风格确认是生成角色定妆图前的硬门槛：分支可视化图确认后，必须先问用户希望角色定妆图生成什么风格，再生成角色图。默认提供 `抖音仔仔`、`真人`、`日本动漫`、`其他` 等选项；用户选择 `抖音仔仔` 时，默认使用 skill 内置参考图提取风格；如果用户额外提供参考图，则以用户新图为准。

如果 `crate doctor --json` 报未登录，优先运行 `crate auth login`。如果浏览器未弹出或登录卡住，按 Crate 指南先运行：

```bash
bytedcli --site i18n-tt auth login
```

扫码成功后再运行 `crate auth login`。在 Codex 沙箱或登录窗口无法正常返回时，可以用已登录的 ByteCloud JWT 作为当前命令环境变量：

```bash
export CRATE_TOKEN=$(bytedcli --json --site i18n-tt auth get-bytecloud-jwt-token | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['jwt'])")
crate doctor --json
```

不要把 JWT 打印给用户或写入文件。

## 快速输入

优先从用户消息中提取以下信息：

- 背景/主题/设定
- 目标观众
- 主角是谁
- 主角最想要什么
- 用户幻想关键词（例如：复仇、逆袭、被爱、权力、财富）
- 分支数量
- 单段时长
- 画幅
- 视觉风格
- 主要角色数量
- 结尾情绪
- 禁忌或不要写的内容

如果信息不足，最多追问 3 个关键问题：观众幻想、主角目标、主角外观偏好。若用户要求快速开始，默认：

- 分支结构：每集默认 2 个选择节点，形态为 `ROOT -> Choice 1A/1B -> Mid A/B -> Choice 2A/2B -> Ending A/B/C/D`
- 角色图：先生成主角 1 张定妆图；若有明确反派，再生成反派 1 张
- 单段时长：5 秒
- 画幅：9:16
- 生图模型：使用 ChatGPT/OpenAI 图像生成，默认走 `imagegen` skill 的内置 `image_gen` 工具
- 分支可视化：分支树生成后默认生成 1 张思维导图/流程图风格可视化图，保存到 `<output_dir>/branch_map.png`
- 视频 provider：默认优先性价比路线 `fal.ai` 或 `Replicate`；若用户已指定 Crate、Kling、Runway、Luma 或本地 ComfyUI，则按用户指定路线。
- 视频模型：默认先用低成本图生视频模型跑 3 条试片（ROOT、一个中段、一个最佳结局），再决定是否全量生成或用更高质量模型重跑关键节点。
- 输出目录：`interactive-shortdrama-output/<slug>/`

## 题材选择

剧本生成前必须先完成题材选择。题材选择必须优先拉起可交互按钮；如果运行环境提供多选按钮、chips、`request_user_input`、`AskUserQuestion` 或等价选项控件，必须用它来呈现题材选择。按钮不可用时，才降级为文本列表。

题材按三个维度划分，允许多选：

- 背景：故事发生的时空、社会环境或空间场域。
- 主题：故事主类型、情绪方向或叙事母题。
- 设定：更具体的爽点机制、人物关系、身份机制或金手指。

按钮 label 规则：

- 按钮 label 使用标签本身，不放序号、冒号、英文、解释或长句。
- 优先使用短标签；平台原始标签超过四字时可以保留，例如 `女性成长`、`追妻火葬场`、`强者回归`。
- 如果交互工具支持 description，可把解释放 description；如果不支持，把解释放在按钮组前后的正文里。
- 自定义入口使用 `自定灵感`。

如果用户没有明确题材：拉起三组多选按钮，分别是背景、主题、设定。允许用户每组不选、单选或多选；至少选择一个标签，或直接回复一句灵感。

如果用户给了故事种子但没有明确题材：先从种子中推荐少量默认勾选标签，再让用户增删。例如“被家族抛弃的女孩，三年后带新身份回来复仇”可推荐：背景 `现代`、`都市`；主题 `现言`、`女性成长`；设定 `打脸虐渣`、`大女主`、`马甲`、`豪门`、`强者回归`。

如果用户已经明确题材：把用户输入归入背景、主题、设定三个维度，简短复述并请用户确认；用户确认后再进入故事梗概和故事评估维度。

## 多 Agent 协作

只有当用户明确要求“多 agent / 并行 / 子 agent / 分工协作”时，才真实 spawn 子 agent。否则在主 agent 内按同样角色完成。

默认并行分工：

1. **剧情架构 agent**：提炼用户幻想关键词、故事梗概、主角目标、核心冲突、人物成长。
2. **互动设计 agent**：设计分支节点、选择后果、分支差异和结局回收。
3. **角色定妆 agent**：设计主要角色外观、服装、表情、记忆点和 ChatGPT/OpenAI 生图 prompt。
4. **剧本 agent**：按节点写短剧场景、动作和对白。
5. **视频导演 agent**：基于确认后的角色图，统一场景视觉、动作调度和视频生成 prompt，并按 provider 能力适配参数。
6. **质量审查 agent**：按剧情标准、互动标准、角色一致性和视频可生成性审查。

主 agent 负责合并所有产物，去重、修正冲突，并生成最终可执行清单。

## 剧情质量标准

每个故事必须满足：

1. **目标明确**：主角要什么、为什么现在必须要。
2. **冲突升级**：主线阻力层层递进，不能平铺。
3. **因果成立**：每个关键事件都有前因后果。
4. **情绪张力**：情绪有压迫、反击、失控、释放。
5. **节奏得当**：短场景、强转折、少闲笔。
6. **人设鲜明**：主要人物有欲望、手段、缺陷、记忆点。
7. **人物成长**：结尾时主角已经不是原来的人。

互动短剧还必须满足：

- 每个选择都有代价或误导。
- 每集默认设置 2 个选择节点；用户明确要求单选择或线性剧情时才减少。
- 第二个选择节点必须改变过程、代价或真相获取方式，不能只把同一过程换成不同结局。
- 分支不是只换结局，过程也要发生变化。
- 分支结局必须回收主角目标和成长。
- 用户能一眼看懂每个选择的诱惑点。

## 输出结构

先完成题材选择，再输出故事梗概、故事评估维度、分支树和分支可视化生图清单；生成分支可视化图并给用户确认后，先让用户选择或提供角色图风格，再生成角色图并给用户确认；确认后再输出视频 provider 建议、视频生成清单并执行。

### 3. 分支树

```markdown
## 分支树
- N01 ROOT：开场钩子
  - Choice 1A：选择文案 -> N02
  - Choice 1B：选择文案 -> N03
- N02：第一分支 A 发展
  - Choice 2A：选择文案 -> N04 Ending A
  - Choice 2B：选择文案 -> N05 Ending B
- N03：第一分支 B 发展
  - Choice 2C：选择文案 -> N06 Ending C
  - Choice 2D：选择文案 -> N07 Ending D
```

### 4. 分支可视化图生成清单

分支树生成后，生成一条分支可视化生图记录：

```json
{
  "asset_id": "B01",
  "type": "branch_map",
  "ratio": "16:9",
  "generator": "chatgpt_imagegen",
  "output": "interactive-shortdrama-output/project/branch_map.png",
  "prompt": "..."
}
```

分支可视化 prompt 必须包含：剧名、题材、ROOT 节点、两层选择节点、每个选择文案、每个分支节点、结局标题、分支因果关系、清晰信息层级、无人物写实脸、无水印。

默认视觉形态：

- 横版 16:9。
- 思维导图或流程图画板风格，所有节点按从左到右的时间线/流程图方向排列。
- ROOT 放在最左侧，第一层分支在中左，第二层分支在中右，4 个结局在最右侧；不要使用中心向左右发散的布局。
- ROOT 与其他剧情节点必须保持同一套视觉风格、卡片样式、字号层级和边框语言，只能通过 `ROOT` 标签或位置区分，不要把 ROOT 做成徽章、光效核心、异形图标或另一种画风。
- 必须把“剧情节点”和“选择”视觉区分开：剧情节点使用实线卡片/节点；选择不要画成独立剧情卡片，优先写在连接线上，或作为贴在线上的虚线/小标签；可用虚线表示选择路径。
- 用不同颜色区分分支，但避免花哨影响可读性。
- 每个节点文字短，优先 8-14 个中文字符；不要把整段剧本塞进图里。

如果生图模型生成的中文文字不稳定，仍然生成视觉版分支图，但同时保留 Markdown 分支树作为权威文本；不要把图片里的文字当作唯一事实来源。

### 5. 节点剧本

```markdown
## N01 | 内景/外景 · 地点 | 时间段

场景动作描写。

角色名：（状态/动作）台词内容

选择点：
A. 选择文案
B. 选择文案
```

不要写景别、角度、运镜；这些属于分镜拆解。视频 prompt 可以写视觉和动作，但不要把镜头术语堆满。

### 6. 角色图风格确认

分支可视化图确认后，生成角色定妆图之前，必须先问用户一个问题：

```markdown
角色定妆图想要什么风格？

- 抖音仔仔
- 真人
- 日本动漫
- 其他
```

如果运行环境提供按钮、chips、`request_user_input` 或等价选项控件，优先用按钮呈现；按钮不可用时，降级为文本列表。

选项规则：

- `真人`：生成写实影视剧定妆照风格，适合后续真人短剧视频。
- `日本动漫`：生成日系动画角色设定图风格，适合二次元或动画短剧。
- `抖音仔仔`：默认参考 skill 内置图片 `assets/douyin-zaizai-style/douyin_zaizai_front_smile.png` 和 `assets/douyin-zaizai-style/douyin_zaizai_three_quarter.png`；如果用户另给参考图，则以用户新图为准。先提取参考图的风格特征，例如头身比、脸型比例、五官简化方式、材质、线条、色彩、光照和整体质感，再把这些风格特征写入每个角色图 prompt。不要照搬参考图中的具体人物身份、商标、水印、服装、配饰或受版权保护的角色。默认风格定义：软萌 3D 卡通潮玩人偶，约 `2.8 头身`，可接受范围 `2.5-2.8 头身`，头部约占全身高度 `35%-40%`，大头小身体，圆润脸型，短小四肢，低龄化比例，光滑搪胶/软塑质感，柔和棚拍光，干净浅色背景，边缘圆润，简化五官，可爱但带轻微酷感。
- `其他`：让用户用一句话描述风格；如果描述过于模糊，再用最多 1 个问题追问。

角色图风格一旦确认，后续所有角色图 prompt 必须使用同一种风格，以保证角色资产统一。

### 7. ChatGPT 角色定妆图生成清单

每个主要角色先生成一条 ChatGPT/OpenAI 生图记录：

```json
{
  "character_id": "C01",
  "name": "林晚 / 沈知意",
  "role": "protagonist",
  "ratio": "9:16",
  "generator": "chatgpt_imagegen",
  "output": "interactive-shortdrama-output/project/characters/C01_linxia.png",
  "prompt": "..."
}
```

默认先生成：

- 主角定妆图：必须生成并确认。
- 核心反派定妆图：当反派会在多个视频节点出现时生成。
- 其他角色：仅当剧情强依赖外观一致性时生成。

角色图 prompt 必须包含：年龄、性别、地域/气质、发型、五官记忆点、服装、表情、身份感、画幅、用户确认的角色图风格、禁止字幕水印。主角必须有 1-2 个稳定记忆点，例如泪痣、发型、耳饰、服装色系。

如果用户确认的角色图风格是 `抖音仔仔`，角色图 prompt 必须显式写入：`约 2.8 头身，可接受范围 2.5-2.8 头身，头部约占全身高度 35%-40%`。

角色定妆图默认必须生成“多视图角色设定板”，不要只生成单张站姿照。画面至少包含一个完整正面全身主视图，并尽量包含侧面、背面、半身表情特写和关键记忆点细节小图。多视图设定板用于后续视频生成保持人物一致性；如果用户明确要求“只要单张站姿照”时才可以改成单图。

使用 `imagegen` 时：

- 默认调用内置 `image_gen` 工具，不需要 `OPENAI_API_KEY`。
- 如果用户明确要求 CLI/API/model 路径，才按 `imagegen` skill 的 fallback CLI 规则处理。
- 角色图是项目后续视频生成的引用资产，生成后必须保存到当前工作区输出目录，不要只保留在 `$CODEX_HOME/generated_images/...`。
- 若一次生成多个角色，每个角色单独生成，避免不同角色互相污染。

### 8. 用户确认点

分支可视化图生成后必须先展示给用户，并明确询问分支结构是否确认。

- 用户确认分支：进入角色图风格确认。
- 用户要求调整分支：先修改分支树，再重生成分支可视化图。
- 用户未确认分支：不要生成角色定妆图。

角色图风格确认后才能生成角色定妆图。

- 用户选择 `真人` / `日本动漫` / 其他明确风格：进入角色定妆图生成。
- 用户选择 `抖音仔仔`：优先使用内置参考图提取风格；如果用户提供了新参考图，改用用户新图提取风格；随后进入角色定妆图生成。
- 用户未确认风格：不要生成角色定妆图。

角色图生成后也必须展示给用户，并明确询问是否确认这些角色外观。

- 用户确认：进入视频生成清单。
- 用户要求调整：只重生成需要调整的角色图。
- 用户未确认：不要提交视频生成任务。

### 9. 视频生成清单

每个节点生成一条记录：

```json
{
  "node_id": "N01",
  "role": "root",
  "duration": 5,
  "ratio": "9:16",
  "provider": "fal.ai / replicate / crate / kling / runway / luma / local",
  "model": "image_to_video / wan / hailuo / kling / ...",
  "character_refs": ["interactive-shortdrama-output/project/characters/C01_linxia.png"],
  "character_ref_urls": ["https://..."],
  "output": "interactive-shortdrama-output/project/N01_root.mp4",
  "prompt": "..."
}
```

视频生成清单面向最终视频模型时，默认只保留一段完整 `prompt`，不要把同一段剧情拆成过多字段。`prompt` 必须从 `节点剧本` 继承场景动作、对白动作和对白，不得只保留画面描述而丢失台词或人物说话时的状态/动作。

`prompt` 必须显式引用已确认角色图中的外观特征，保持人物连续；可以压缩剧情动作，但不能改变 story_plan 里的剧情事实。写法建议为一整段自然语言：场景和时间 + 人物外观记忆点 + 场景动作 + 角色说台词时的动作/状态 + 台词内容 + 氛围风格 + 禁止字幕水印。

选择点和互动关系不要塞进视频生成 prompt，除非明确要在画面里做选择按钮或字幕覆盖；默认由 `manifest.json` 记录。

如果后续需要配音、字幕、口型或剪辑分轨，再额外导出 `dialogue` 或 `script_beats` 辅助文件，但不要让默认视频生成清单变重。

## 执行流程

1. 先让用户选择或确认题材；用户未选择或确认前，不要写正式剧本。
2. 题材确认后，输出故事梗概、故事评估维度和分支树。
3. 用 ChatGPT/OpenAI 图像生成分支可视化图，并保存到 `<output_dir>/branch_map.png`。
4. 展示分支可视化图给用户确认；用户未确认前，不要生成角色定妆图。
5. 分支确认后，输出节点剧本，并询问角色定妆图风格。
6. 用户选择角色图风格；如果选择 `抖音仔仔`，默认使用内置参考图提取风格；如果用户提供新参考图，则以用户新图为准。
7. 风格确认后，输出角色生图清单。
8. 用 ChatGPT/OpenAI 图像生成角色定妆图，并保存到 `<output_dir>/characters/`。
9. 展示角色图给用户确认；用户未确认前停止。
10. 角色确认后，选择视频 provider 和模型；如果用户未指定，按性价比优先推荐 MiniMax Token Plan、fal.ai、Replicate 等路线。
11. 若本地角色图要作为视频首帧或参考图，按所选 provider 的上传/输入要求处理。使用 Crate 时可先上传到 Crate：

```bash
crate upload file "<local_character_image>" --json
```

保存返回的 URL 到视频生成清单的 `character_ref_urls`。

12. 选择视频模型。

用户指定 `seedance2` / Seedance 2.0 时：先提交 1 条最小测试任务；如果返回 `Model seedance2 not found`，立即停止并说明这是服务端模型开关/白名单问题，不能继续消耗尝试。

用户没有指定模型时：优先用 `image_to_video`，并使用上传后的角色图 URL 作为输入参数。

如果某模型无法用 CLI 子命令传入图参数：允许直接调用 Crate BFF，但必须沿用 `crate doctor` 验证过的 base URL 和 JWT，不要打印 JWT。

13. 为每个节点提交视频任务，默认小批量执行，避免一次性消耗过多额度。
14. 每条提交必须记录 `taskId`、输出文件路径、模型、参数和使用的角色参考图 URL。
15. 若使用异步任务，按 provider 的查询接口轮询。
16. 若失败，记录错误原因，修正 prompt 后再询问是否重试。
17. 完成后输出分支可视化图、角色图、视频文件列表和 `manifest.json` 路径。

默认 Crate 异步提交命令形态：

```bash
crate generate submit --model image_to_video --prompt "<prompt>" --json
```

注意：当前 `crate generate submit` 的 CLI 实现可能不读取 `--params`。如果需要传角色图 URL、首帧图、比例、时长等参数，应直接请求 BFF：

```bash
export CRATE_TOKEN=$(bytedcli --json --site i18n-tt auth get-bytecloud-jwt-token | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['jwt'])")
node submit-crate-video.js
```

`submit-crate-video.js` 应发送：

```json
{
  "modelId": "image_to_video",
  "parameters": {
    "prompt": "...",
    "imageUrls": ["<uploaded_character_or_first_frame_url>"],
    "aspectRatio": "9:16",
    "duration": 5
  }
}
```

请求头使用 `x-jwt-token: <TOKEN>`，接口为 `<crate_base_url>/v1/generation/submit`。如果返回任务 ID，用 `crate task get <taskId> --json` 轮询；如果返回 `task_status: 3` 且无错误原因，记录原始返回并停止，不要盲目重复提交。

Seedance 2.0 专项规则：

- Crate CLI `model list` 中可能静态显示 `seedance2`，但这不代表服务端提交接口已开放。
- 调用 `seedance2` 前必须先做最小提交验证：

```bash
crate generate submit --model seedance2 --prompt "short vertical test video" --json
```

- 如果返回 `Model seedance2 not found`，结论是当前账号/环境服务端未开放该模型；不要继续换参数重试，改为询问用户是否切换 `image_to_video`，或让用户联系 Crate 平台确认真实可提交 `modelId` / 白名单。
- 若用户坚持 Seedance，只能在获得可用模型 ID 或权限开通后继续。

## 自检

交付前检查：

- 七项剧情标准是否全部成立。
- 剧本生成前是否已经让用户选择或确认题材。
- 分支选择是否清楚、有诱惑、有后果。
- 分支可视化图是否已生成、展示，并以 Markdown 分支树作为权威文本校对。
- 角色定妆图是否已展示并获得用户确认。
- 视频 prompt 是否继承已确认角色图的稳定外观特征。
- 每个节点 prompt 是否只描述一个清晰动作段落。
- 角色外观、服装、地点、情绪是否连续。
- 每个提交任务是否有 `taskId` 或完成输出、状态、输出路径。
- Crate 失败时是否记录了模型、参数、taskId、`task_status` 和错误信息；遇到 `Model ... not found` 时是否停止而不是盲目重试。
- `manifest.json` 的节点和视频文件是否一一对应。
